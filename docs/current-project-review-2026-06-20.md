# 当前项目审查记录 2026-06-20

## 当前结论

项目已经具备私有会员媒体站的主链路：邀请码注册、登录、会员等级、管理员发布内容、MinIO 私有媒体、Nginx 统一入口和 Docker Compose 单机部署。

但它还不是完整成熟产品。当前最明显短板是上传体验、列表规模化、后台工作流深度、备份运维和内容运营能力。

## 本轮已处理

- MinIO 不再直接映射公网 `9000/9001`，媒体只通过 Nginx `/rinana-media/` 访问。
- 视频上传默认限制统一为 `95MB` 文件、`100MB` 请求体、Nginx `100m`，适配 Cloudflare Free/Pro 代理请求体 `100MB` 上限。
- `/api/media/` 单独关闭 Nginx 请求缓冲和临时文件落盘，减少大文件上传时中间层开销。
- `/rinana-media/` 保留 `Range` / `If-Range`，并返回 `Accept-Ranges: bytes`，支持视频拖动和续播。
- 前端上传校验和后台提示已同步到 `95MB`，提示管理员优先上传 MP4（H.264 + AAC）。
- 首页和卡片封面从 CSS background 改为真实 `img`，浏览器可以做懒加载和异步解码。
- `/api/content` 从“查出全部已发布内容再 Java 过滤”改成按 `status + visibility` 在数据库侧查询。
- 管理后台已从单一大表单调整为总览、发布工作台、媒体状态、内容库、邀请码和用户管理的分区布局。
- 卡片已支持详情链接，新增 `/zh/posts/[id]`、`/zh/albums/[id]`、`/zh/videos/[id]` 基础详情页。
- 生产环境没有可浏览内容时，首页和列表页会显示正式空状态，不再只留下空白内容区。
- 前端卡片和后台内容行增加 `content-visibility: auto`，长列表首屏渲染压力更低。

## 仍然不够的地方

### 界面和布局

- 当前已经比之前更像粉色系会员媒体工具，但品牌资产仍不足，主要靠色彩和排版支撑，还缺真实头像、封面规范和运营视觉。
- 管理后台可用性提升了，但还没有预览、草稿、批量管理和媒体库检索，所以离成熟 CMS 还有距离。
- 移动端可用，但后台长表单和统计卡片仍偏密，后续应继续拆步骤和减少同屏操作量。

### 功能

- 详情页已有基础入口，但还只是客户端从当前数据集中查找，后续应补后端单条详情接口和服务端渲染数据。
- 没有分页、无限滚动或按分类筛选，内容多了以后前台和后台都会变重。
- 视频没有上传进度、暂停/恢复、分片上传、失败重试。
- 没有视频转码、封面自动生成、HLS、多码率，必须依赖管理员上传浏览器可播放 MP4。
- 没有内容草稿、定时发布、置顶、搜索、标签。
- 后台没有审计日志页面，误删和权限调整缺少可追溯入口。

### 性能

- 前端仍由全局客户端状态承载大量数据，详情页也暂时复用这份状态；后续应把公开页面尽量服务端化或按路由拆分状态。
- 内容接口仍一次返回 posts/albums/videos 三类全量结果，下一步需要分页参数。
- 媒体 URL 是短时签名 query，Cloudflare CDN 命中率不会天然很高。
- 上传大于 `95MB` 的视频现在不建议走 Cloudflare 代理，必须另做分片或上传专用入口。

## 下一步优先级

1. 给内容接口加分页：先后端 `limit/cursor`，再前端加载更多。
2. 给详情页补后端单条接口和服务端数据加载，避免依赖全局客户端状态。
3. 给视频上传加进度显示和失败重试；超过 `95MB` 再考虑分片上传。
4. 重做后台发布体验：分步骤、预览、草稿、媒体库。
5. 设计 Cloudflare CDN 规则：只对安全媒体路径配置 Cache Rules，不全站忽略 query string。
6. 增加 PostgreSQL 和 MinIO 自动备份脚本。
7. 继续做视觉第二轮：引入真实品牌素材，降低单色粉系占比，提升内容卡片和后台工具感。
